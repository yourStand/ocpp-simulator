# ガバナンスモデルと今後の論点

## 現在: モデルA（CPO主導ホスト）

ユアスタンドが本リポジトリを単独運営する。各ベンダー領域も当面はユアスタンドが
暫定リファレンスとして著作・管理する。まず社内で価値を実証する段階。

**現在の参加メーカーはユアスタンドのみ**（`vendors/yourstand/` = 自社製 C2盤 YSC2）。
自社製品のためライセンスは CC-BY-4.0 を直接宣言しており、暫定マーカー
`LicenseRef-Provisional-VendorReference` は使っていない。

## 今後議論していくこと

メーカーをどう巻き込み、どんな運営体制にするかは**まだ決まっていない**。
以下は今後、関係者と相談しながら詰める論点。

- **参加方法**: 各メーカーにどの形で参加してもらうか
  （本リポへのディレクトリ単位PR / 各社の別リポ連携 / 中立団体ホストへの移行 など）。
- **ライセンス・所有**: ベンダー領域の正式ライセンスと所有を誰がどう決めるか。
  参加までは暫定リファレンス（`LicenseRef-Provisional-VendorReference`）として扱う。
- **意思決定とレビュー権限**: CODEOWNERS によるディレクトリ単位の承認委譲をどう運用するか。

下記の選択肢・設計は「どの結論にも動けるようにしておく」ための準備であり、
特定の移行を確定したものではない。

## 選択肢（3案）

| | モデル | 中身 | 長所 | 短所 |
|---|---|---|---|---|
| A | CPO主導ホスト | ユアスタンド運営、各ベンダーが自社ディレクトリをPR | 立ち上げが早い・小回り | 参加抵抗・中立性 |
| B | 業界団体ホスト | 中立団体(OCPP-Japan等)が運営 | 中立性・持続性 | 立ち上げ遅い・意思決定遅い |
| C | マルチリモート | 各ベンダーが自社リポで公開、CPOがsubmodule参照 | ベンダー独立性 最大 | 横断検索/版管理が複雑 |

## どの結論にも動けるようにするための設計上の蝶番

1. **ベンダー軸を最上位で物理分離**（`vendors/<vendor>/`）。
   C移行時は当該ディレクトリを `git subtree split` で独立リポジトリへ切り出すだけ。
2. **ライセンス宣言をベンダーごとに分散**（`vendors/<vendor>/REUSE.toml`）。
   ライセンスがディレクトリと一緒に移動する。ルートREUSE.tomlは vendors/ を触らない。
3. **CODEOWNERS をベンダー単位で分割済み**。委譲は1行の差し替え。
4. **`vendors/_template/`** で新規ベンダーonboardingをコピー操作に落とす。
5. **samples と vendors の境界を明示**。共通の参考サンプル(ホスト管理)と機種固有(各社所有)
   を混ぜない。

## 議論のきっかけになりそうな状況（参考）

確定事項ではなく、こうなったら方式を見直す・議論する、という目安。

- 2社目のメーカーが参加 → CODEOWNERS によるディレクトリ単位の委譲運用を議論。
- メーカーが「自社リポで持ちたい」 → C（submodule）方式を検討。
  横断回帰テストを維持できるかが論点（Cの弱点）。
- 業界での中立運営要求が高まる → B（団体ホスト）を検討。CLA等のガバナンスをどう置くか。

## samples/ における OCA スキーマの扱い（重要）

`samples/` 配下に Open Charge Alliance 配布の OCPP スキーマ原本を verbatim で取り込んで
はならない。OCA の利用条件に従い、ユアスタンドが自由に再ライセンスすることはできない。

取り込みが必要な場合は:
- `samples/` の CC-BY-4.0 宣言の対象から除外し、OCA条件を明示した別パスに隔離する。
- 取り込み前に OCA の最新条件を確認する（OCPP 1.6 と 2.0.1 で条件が異なる可能性）。
